01
Event-driven trigger · FR-02
Trigger & DRI Assignment
OMS disconnect event or CSR manual escalation initiates the pipeline. DRI named before any output is produced.
›
Pattern
Event-driven trigger with dual entry points. Trigger A (proactive): OMS disconnect event fires automatically before the next CC&B billing run. Trigger B (reactive): CSR or DRI manually initiates from the escalation screen when a dispute has already been filed.
Purpose
Close the window in which an invalid bill can be generated. Proactive trigger eliminates the dispute before it exists. Reactive trigger replaces the 45–90 min manual CSR investigation with an automated pipeline.
Input
  • Trigger A: OMS disconnect event payload — account ID, disconnect timestamp, work order type
  • Trigger B: CSR escalation form — account ID, dispute type, notes
Output
  • Escalation ID created and logged to audit trail
  • DRI assigned and identity stamped on all subsequent outputs (FR-10, BR-01)
  • Pipeline state: Initiated
  • Ticket status: Analysing — visible to CSR and DRI dashboards
Business rules
BR-01 DRI named before any output FR-02 Trigger before next billing run FR-10 DRI identity on all outputs
NFR target
Immediate — trigger latency must not delay billing run suppression
02
Parallel fan-out retrieval · FR-01
5-System Parallel Ingestion
Agent queries all 5 enterprise systems simultaneously. Every field extracted is stamped with source system and timestamp for the audit trail.
›
Pattern
Parallel fan-out retrieval. All 5 system queries are issued concurrently, not sequentially. Results are collected and normalised into a unified evidence object before the next step begins. No system is treated as authoritative over another (BR-02).
Purpose
Assemble the complete cross-system evidence base in a single pass. Replace what previously required 4–6 separate manual data pulls across IT, Ops, and Finance teams taking 4–6 hours collectively.
Input per system
  • CC&B: Account status · billing period start/end · billing rules applied · bill generation timestamp
  • MDM: Meter read type (actual vs estimated) · reads by date · consumption values mapped to billing period
  • OMS: Work order type · disconnect/reconnect timestamps · field technician confirmation records
  • CRM: Prior complaint history · complaint type · billing-hold flags · manual account notes
  • GL: Revenue posting status · period of posting · reversal implications
Output
  • Unified evidence object — all 5 system payloads normalised and timestamped
  • Every field written to audit trail with source system + extraction timestamp (FR-09)
  • Pipeline state: Evidence assembled
Business rules
BR-02 CC&B never sole input FR-01 All 5 systems required FR-09 100% audit citation
NFR target
<7 min total — ingestion contributes the majority of the end-to-end assembly window (NFR-01)
03
Temporal cross-reference · FR-03
Reconciliation — Period Classification
OMS disconnect timestamp mapped against CC&B billing period. Every day classified. MDM reads mapped to the same breakdown. The evidence base is reconciled before conflicts are checked.
›
Pattern
Temporal cross-reference and day-level classification. The agent aligns three independent time axes — CC&B billing period boundaries, OMS disconnect timestamp, and MDM read dates — into a single reconciled timeline. Every day in the billing period is assigned exactly one classification. This is the core analytical step from which all conflict detection follows.
Purpose
Establish the ground truth of what was billed vs what was physically delivered. Without reconciliation, conflict detection would compare raw system outputs that speak different time grammars. Reconciliation creates the shared reference frame.
Input
  • CC&B billing period: start date, end date, total billed consumption
  • OMS disconnect timestamp + field tech confirmation record
  • MDM meter reads per date — flagged actual vs estimated
Output
  • Day-level billing period map: each day labelled Pre-disconnect (billable), Post-disconnect (non-billable), or Disconnect day (proration required)
  • MDM reads mapped to the same day-level breakdown — actual reads vs estimated reads per classification bucket
  • Billing overlap confirmed or denied — input to conflict detection
  • Reconciliation object written to audit trail (FR-09)
Business rules
FR-03 Day-level classification required BR-03 Estimated reads on post-disconnect days = phantom consumption
NFR target
Within <7 min — runs within the overall context assembly window (NFR-01)
04
Rule-based cross-system validation · FR-04
Conflict Detection
4 architectural conflict checks run against the reconciled evidence base. Every conflict is surfaced explicitly — never silently resolved. Completes in <15 sec.
›
Pattern
Rule-based cross-system validation against reconciled evidence. Four deterministic checks are applied to the reconciled evidence object. Each check compares a specific pair of systems against a defined invariant. A conflict is raised if the invariant is violated. No probabilistic scoring — conflicts are binary: present or absent. BR-07 prohibits silent resolution.
Purpose
Surface the architectural root causes that produced the invalid bill. Each conflict maps to a specific system integration gap that engineering must close. The conflict list is the input to both the recommendation engine and the engineering Jira handoff.
Input
  • Reconciled period map (from step 3)
  • Full unified evidence object (from step 2)
4 conflict checks
C1 · OMS ↔ CC&B — Full-cycle bill generated despite mid-cycle disconnect. Invariant: if OMS disconnect timestamp falls within CC&B billing period, bill must be prorated or suppressed.
C2 · MDM ↔ OMS — Estimated reads applied to post-disconnect days. Invariant: MDM read type for any post-disconnect day must be zero consumption, not estimated historical average (BR-03).
C3 · CRM ↔ CC&B — Prior complaint on record with no billing-hold propagated. Invariant: any CRM complaint flag on a billing account must propagate a billing-hold to CC&B (BR-04).
C4 · GL ↔ CRM — Revenue posted for a bill under formal dispute. Invariant: GL revenue should not be finalised while a CRM dispute is active (BR-06).
Output
  • Conflict manifest — list of all triggered conflicts with systems named, invariant violated, and evidence cited
  • Each conflict written to audit trail with source citations (FR-09)
  • Confidence signal raised if any system data is missing or estimated (FR-06)
  • Pipeline state: Conflicts detected
Business rules
BR-07 Never silently resolve BR-03 Phantom consumption flag BR-04 Billing-hold propagation BR-06 GL reversal flag
NFR target
<15 seconds — conflict detection only (NFR-01). Runs within the overall <7 min assembly window.
05
LLM reasoning + structured assembly · FR-05, FR-06
Recommendation & Decision-Ready Brief
Agent maps each conflict to a recommended action, then assembles the full cited brief. 100% citation coverage required. Readable by regulator, VP, or DRI without additional context.
›
Pattern
LLM reasoning over structured evidence + deterministic brief assembly. Two sub-steps: (a) the LLM maps each detected conflict to the correct recommended action using the FR-05 scenario table — every recommendation cites the source system; (b) the brief assembler stitches confirmed facts, period breakdown, conflicts, and recommendations into the FR-06 structured format with 100% citation coverage.
Purpose
Produce a single document from which the DRI can make an informed, defensible decision in under 5 minutes without querying any system directly. Replace the 2–4 hour manual summary that was unstructured, uncited, and author-dependent.
Input
  • Conflict manifest (from step 4)
  • Unified evidence object (from step 2)
  • Reconciled period map (from step 3)
  • FR-05 scenario → action mapping table
Recommendation scenarios (FR-05)
C1 → Suppress post-disconnect portion · issue prorated credit
C2 → Flag phantom consumption · adjust MDM to zero for non-billable days
C3 → Link CRM complaint · apply billing-hold flag · trigger review
C4 → Generate credit note · flag GL for revenue adjustment · regulatory filing
Output
  • Decision-ready context brief — confirmed facts, period breakdown, conflicts, recommendations, confidence signals (FR-06)
  • 100% citation coverage — every claim cites source system + timestamp (FR-09, NFR-02)
  • Confidence signal included where data is incomplete or reads cannot be verified
  • Brief written to audit trail · pipeline state: Awaiting DRI
  • DRI and CSR dashboards updated with notification
Business rules
FR-05 Recommendation per scenario FR-06 Structured brief format FR-09 100% citations NFR-02 Audit completeness
NFR target
<60 seconds — brief generation within the overall <7 min assembly window (NFR-01)
06
Human-in-the-loop gate · FR-08, FR-09, FR-10
DRI Approval Gate
Mandatory decision-authority gate. DRI reviews the brief, approves or rejects with logged reason. No output generated or dispatched without this gate. Logged to immutable audit trail.
›
Pattern
Human-in-the-loop approval gate. The pipeline is paused at this step until explicit DRI action is taken. This is not a notification — it is a hard gate. The agent cannot proceed to step 7 without a DRI decision logged with timestamp and approver identity. Approve, reject with reason, or flag as exception — all three paths are valid and all three are logged.
Purpose
Ensure human judgment is applied at the single point where consequential action is authorised. The agent does all the analytical work; the DRI holds the authority. This separation is both an ethical requirement and a regulatory one — a human must be accountable for every credit note, proration adjustment, and regulatory submission.
Input
  • Decision-ready context brief (from step 5)
  • DRI identity (assigned at step 1)
  • Confidence signal from brief (flags cases warranting extra scrutiny)
DRI decision paths
Approve — pipeline proceeds to step 7. The 3 outputs are generated as drafts for CSR review (not auto-dispatched). Approval logged with timestamp + DRI identity.
Reject — mandatory reason field required. Ticket returned to CSR with rejection note. Rejection logged with timestamp, DRI identity, and reason.
Flag exception — case escalated for manual review. Logged with timestamp and exception rationale. Pipeline paused pending resolution.
Output
  • Approval / rejection / exception decision logged to immutable audit trail with timestamp + DRI identity (FR-09)
  • On approve: step 7 triggered — 3 output drafts generated, pipeline state → Awaiting CSR output review
  • On reject: pipeline state → Rejected, CSR notified with reason
  • On exception: pipeline state → Escalated, Analyst Manager notified
Business rules
FR-08 HITL required BR-05 No dispatch without DRI FR-10 DRI identity on outputs FR-09 Approval logged
NFR target
<5 min DRI decision — requires brief to be actionable without further research (NFR-01)
07
Single-source multi-format generation + CSR review gate · FR-07, BR-08
3 Outputs — CSR Review & Dispatch
Customer summary, executive brief, and engineering Jira handoff generated from the same evidence base as drafts. The CSR reviews, edits, and approves each one before it is sent. Divergent narratives structurally impossible. Audit trail finalised.
›
Pattern
Single-source multi-format generation. All three outputs are rendered from the same reconciliation evidence object using audience-specific formatting templates. BR-08 makes divergent narratives structurally impossible — the regulatory response and customer summary are generated from identical facts. The agent does not write three independent documents; it renders one evidence object three ways.
Purpose
Eliminate the ~30% regulatory inconsistency rate produced by the manual process where Legal and CSR wrote independent documents from the same imprecise notes. Every stakeholder receives the same facts in the format appropriate to their role and action.
Input
  • Approved reconciliation evidence object (from step 5)
  • DRI approval record with timestamp and identity (from step 6)
  • Three audience-specific format templates (FR-07)
3 outputs (FR-07)
Output 1 · Customer · <2 min read
Customer Summary Letter — What we found · What we have done · What happens next · Next update date. Plain language, non-technical. No jargon. No system names.
Output 2 · Executive · 1 page
Executive Brief — Situation · Financial impact · Findings · Next steps. GL reversal implications included. Regulatory exposure stated. DRI identity and approval timestamp on cover.
Output 3 · Engineering · Jira story
Engineering Handoff BILL-NEW-1147 — P1 priority · Linked systems (OMS-DISCON, MDM-EST-READS) · 6 acceptance criteria for CC&B architectural fix · Staged deploy with 4-week parallel validation.
CSR review gate (per output)
Each of the 3 outputs is created in a pending-review state. The CSR opens each draft, edits the content if needed, and approves it. Nothing is sent until all 3 are approved — this is a second human-in-the-loop gate, complementing the DRI decision gate at step 6. Every approval (and any edit) is written to the immutable audit trail with the CSR’s identity and timestamp.
Review & approve — CSR approves the draft as-is or after edits. Per-output approval logged (FR-09).
Dispatch on completion — once all 3 are approved, outputs are sent and the ticket resolves.
Output
  • 3 draft outputs generated — ticket status: Awaiting CSR output review
  • CSR reviews, edits and approves each output individually — one immutable audit entry per approval (FR-09)
  • On all 3 approved: outputs dispatched to respective audiences · final dispatch record written
  • Ticket status: Resolved — visible across all dashboards
  • Analyst Manager execution log updated with completion record
  • First coherent stakeholder update delivered within <24 hrs of escalation (NFR-01)
